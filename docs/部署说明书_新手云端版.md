# AMA-RT 云端部署说明书（新手照抄版）

> 这份说明书的目标：**你完全不懂编程，只要按顺序把每一行命令复制粘贴到云服务器里，就能把系统跑起来。**
>
> 适用系统：**Linux（推荐 Ubuntu 22.04 / 24.04 64 位）**。
> 适用项目：**AMA-RT 妖币右尾捕获系统（本仓库）**。
>
> 你会得到三种用法：
> 1. **空盘跑（LIVE_SHADOW）—— 半自动化信号辅助模式**：系统连真实行情，把识别到的候选币、计划入场价、计划止损、计划止盈、计划杠杆、风控理由推到 Telegram 给你人工判断；你可以人工手动下单。**空盘模式系统本身绝不在币安下真实订单。**
> 2. **实盘跑（LIVE_LIMITED）—— 真实资金模式**：真实订单仍必须经过风险决策（LiveRiskDecision）+ 执行网关（LiveExecutionGateway）+ 全部门控，需要层层确认才会真的下单。
> 3. **用 Telegram 指令在“空盘 / 实盘”之间切换**（`/mode live_limited` → `/confirm_live CODE` 武装；`/mode shadow` 切回）。

---

## ⚠️ 开始之前，先读这 8 句话（很重要）

1. 本系统**默认是最安全状态**：空盘（`LIVE_SHADOW`）、不真实下单、不动你的钱。
2. 想真实下单，必须你**亲手**改好几个开关 + 在 Telegram 上做“双重确认握手”，**任何一步没做，系统都不会下单**。
3. **币安 API Key 不要开“提现（Withdraw）”权限。** 本系统永远用不到提现。
4. **API 密钥 / 机器人 Token 绝对不能发给别人，也不能传到 GitHub。** 本说明书会教你放在服务器本地的 `.env` 文件里（已被 `.gitignore` 屏蔽）。
5. 实盘**先用最小金额（10 USDT 档位）测试**，确认一切正常再考虑加大。
6. 加密货币交易**有亏光本金的风险**。本说明书只讲“怎么部署和操作软件”，**不构成任何投资建议**。盈亏由你自己负责。
7. 关于“全自动实盘”：本项目出于安全设计，**真实下单的最后一步仍然需要人工确认指令**（详见第 11 章）。请不要期待“开机就无人值守自动炒币”。
8. 遇到任何看不懂的报错，先翻到最后的 **第 13 章「常见问题排错」**。

## 第 1 章 准备一台云服务器并登录

### 1.1 买服务器
随便选一家云厂商（阿里云、腾讯云、AWS、Vultr、DigitalOcean 等）。配置建议：

- 系统镜像：**Ubuntu 22.04 LTS** 或 **24.04 LTS**（64 位）
- 配置：**2 核 CPU / 4 GB 内存 / 40 GB 硬盘** 起步足够
- 地区：**不要选中国大陆机房**（币安在大陆访问受限），建议选香港、新加坡、日本、美国等
- 记下服务器给你的：**公网 IP**、**用户名（通常是 `root` 或 `ubuntu`）**、**密码或密钥**

### 1.2 登录服务器
在你自己的电脑上打开终端（Windows 用「PowerShell」或「Windows Terminal」；Mac 用「终端」），输入（把 `你的IP` 换成真实 IP）：

```bash
ssh root@你的IP
```

第一次会问 `yes/no`，输入 `yes` 回车；然后输入密码（输密码时屏幕不显示是正常的）。

看到类似 `root@xxxx:~#` 的提示符，就说明**你已经进服务器了**。后面所有命令都在这个窗口里敲。

## 第 6 章 注册 Telegram 机器人，拿到 Token 和你的 Chat ID

你需要两样东西：**机器人 Token** 和 **你的 Chat ID**。

### 6.1 创建机器人，拿 Token
1. 手机 / 电脑打开 Telegram，搜索官方机器人 **@BotFather**。
2. 给它发 `/newbot`。
3. 按提示给机器人起个名字、起个以 `bot` 结尾的用户名。
4. 成功后 BotFather 会发给你一串 **Token**，长得像：
   `123456789:AAFxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`
   **把它复制保存好**，这就是 `AMA_TELEGRAM_BOT_TOKEN`。

### 6.2 拿你自己的 Chat ID
1. 在 Telegram 搜索 **@userinfobot**，给它发任意一句话（比如 `hi`）。
2. 它会回复你一个 **Id**，是一串数字（例如 `7654321`）。
   这就是 `AMA_TELEGRAM_ALLOWED_CHAT_IDS`。

### 6.3 让机器人能给你发消息（关键一步）
**先用你自己的账号，主动给你刚建的机器人发一句 `/start` 或随便一句话。**
（Telegram 规定：机器人不能主动给“没和它说过话”的人发消息。）

> 📌 安全机制提醒：本系统**只听** `AMA_TELEGRAM_ALLOWED_CHAT_IDS` 名单里的人的指令。名单为空 = 谁都不能控制系统（默认拒绝，这是好事）。所以这一步必须填对你的 Chat ID。

## 第 7 章 填写配置文件 .env

### 7.1 复制模板
```bash
cp .env.example .env
```

### 7.2 编辑 .env
用自带的编辑器 `nano` 打开：

```bash
nano .env
```

> nano 操作：用方向键移动光标；改完按 `Ctrl + O` 再回车保存；按 `Ctrl + X` 退出。

把文件里对应的几行填成你的真实值。**先填“空盘版”最小配置**（只接 Telegram 推送，不接交易所私有数据，最安全）：

```ini
# ===== Telegram（必填）=====
AMA_TELEGRAM_BOT_TOKEN=把第6章拿到的Token粘到这里
AMA_TELEGRAM_ALLOWED_CHAT_IDS=把你的ChatID粘到这里
AMA_TELEGRAM_OUTBOUND_ENABLED=true

# ===== 运行模式（空盘，最安全，保持默认）=====
AMA_LIVE_RUNTIME_MODE=LIVE_SHADOW
# 资金档位：两个名字都支持，AMA_LIVE_CAPITAL_PROFILE_ID 优先级更高，建议两个都写成一样
AMA_LIVE_CAPITAL_PROFILE=L0_SHADOW
AMA_LIVE_CAPITAL_PROFILE_ID=L0_SHADOW

# ===== 币安（空盘阶段可以先留空）=====
AMA_BINANCE_API_KEY=
AMA_BINANCE_API_SECRET=
AMA_BINANCE_ENABLE_PRIVATE_READ=false
AMA_BINANCE_ENABLE_PRIVATE_TRADE=false

# ===== 真实下单总开关（先全部关闭）=====
AMA_LIVE_EXCHANGE_LIVE_ORDERS=false
AMA_LIVE_TRADE_AUTHORITY=false
AMA_LIVE_AI_TRADE_AUTHORITY=false
AMA_SECRET_LOGGING_ALLOWED=false
```

> 说明：
> - `true/false` 全部小写。
> - 等号两边**不要加空格**，值也**不要加引号**。
> - `AMA_TELEGRAM_ALLOWED_CHAT_IDS` 如果有多个人，用英文逗号隔开：`111,222`。

保存退出（`Ctrl+O` 回车，`Ctrl+X`）。

### 7.3 锁住文件权限（只让你自己能读）
```bash
chmod 600 .env
```

### 7.4 让命令行能读到 .env 里的变量
本项目脚本是从“系统环境变量”里读密钥的。每次新开窗口运行前，先执行这一句把 `.env` 加载进来：

```bash
set -a; . ./.env; set +a
```

> 这句话的意思是“把 .env 里的每一行都变成环境变量”。**每次重新登录服务器、或新开终端窗口，运行带密钥的命令前都要先敲一遍这句。**（第 12 章用 systemd 常驻时则不用手动敲，systemd 会自动读取。）

## 第 8 章 空盘跑：把信号推到 Telegram（人工判断 / 人工下单）

**LIVE_SHADOW / 空盘跑 是“半自动化信号辅助模式”。** 它的目标是把系统识别到的候选币、计划入场、计划止损、计划止盈、计划杠杆、风险理由推送到 Telegram，供你（operator）人工判断趋势是否正确，并由你决定是否手动下单（或后续切到 LIVE_LIMITED 有资金跑）。**空盘模式下系统本身绝不在币安下真实订单。**

### ⚠️ 先搞清楚：空盘的卡片有三个不同来源（很重要，别误会）

**不要以为“运行 `live_shadow_run.py` 就一定会收到完整的开仓信号”。** 空盘推送到 Telegram 的卡片其实来自三个不同的地方：

| 来源 | 跑什么命令 | 推什么卡 |
| --- | --- | --- |
| **A. 基础状态推送循环** | `live_shadow_run.py --loop --send-telegram` | 账户状态、持仓、PnL、资金费(funding)、资金档位、`LIVE_SHADOW_SUMMARY` 等**状态卡**。用来确认系统在线、API 可读、Telegram 可用、资金状态可见。 |
| **B. 模拟订单 / 模拟执行卡测试** | `live_limited_smoke.py --dry-run --send-telegram ...` | 一张**模拟下单提示卡 / 计划卡 / 拦截卡**，验证“下单卡能不能推到 Telegram”。必须 `real_order=false`。 |
| **C. 策略信号链路（strategy signal chain）** | 由策略链路在产生候选时触发 | 候选币 / 开仓计划 / 风控拒绝卡，例如 `SHADOW_ENTRY_PLAN`、`SHADOW_EXIT_PLAN`、`SHADOW_RISK_REJECT`、`LIVE_RISK_REJECT`、`LIVE_EXECUTION_BLOCKED`。这些卡用于人工观察趋势、判断策略质量。 |

> 一句话记住：**`live_shadow_run.py` 负责基础状态卡；模拟订单卡由 `live_limited_smoke.py --dry-run --send-telegram` 触发；策略候选卡（开仓计划 / 风控拒绝）由策略信号链路触发。** 所以只跑 `live_shadow_run.py` 时，你看到的主要是状态卡，不一定会有开仓候选卡——这很正常。

### 8.1 先发一条测试卡片，确认 Telegram 通了
（先加载 .env，再发送测试）

```bash
set -a; . ./.env; set +a
.venv/bin/python scripts/live_telegram_operator.py --send-test
```

✅ 成功：你的 Telegram 会收到一张机器人发来的测试卡片。
❌ 如果显示 `suppressed`：说明 `AMA_TELEGRAM_OUTBOUND_ENABLED` 不是 `true`，或者你没先给机器人发过 `/start`（回第 6.3 节）。

### 8.2 来源 A：单跑一次基础状态推送（看效果）
```bash
.venv/bin/python scripts/live_shadow_run.py --once --send-telegram
```

这会：读一次真实公开行情（接了私有读还会读你的账户/持仓）→ 生成**状态卡**（账户、持仓、PnL、funding、资金档位、`LIVE_SHADOW_SUMMARY`）→ 推到你 Telegram。每张卡都标注 `real_order=false` / `no_real_order_sent=true`。

> 📌 想让状态卡有账户/持仓/PnL 内容，需要接“币安私有读”：在 `.env` 填一把**只读、无提现权限**的币安 API Key（`AMA_BINANCE_API_KEY` / `AMA_BINANCE_API_SECRET`），并设 `AMA_BINANCE_ENABLE_PRIVATE_READ=true`。即便如此，空盘模式**依然不会下单**。怎么建币安 Key 见第 11.1 节。
>
> 想额外带一份 DeepSeek 智能简报（仅情报、无交易权），加 `--with-ai-briefing`。

### 8.3 来源 B：测试一张“模拟订单 / 计划卡”能否推到 Telegram
```bash
.venv/bin/python scripts/live_limited_smoke.py \
  --dry-run \
  --send-telegram \
  --symbol RAVEUSDT \
  --notional 1 \
  --leverage 1 \
  --planned-entry-price 0.5 \
  --planned-stop-price 0.45 \
  --planned-take-profit-price 0.7 \
  --json
```

说明：
- 这**不会真实下单**。
- Telegram 应收到一张 `real_order=false` 的模拟订单 / 拦截 / 计划卡。
- 真实成交字段为空白：`order_id=--`、`actual_entry_price=--`、`actual_exit_price=--`、`real_capital_changed=false`。

### 8.4 来源 C：让基础状态循环持续跑（每 60 秒一轮）
```bash
.venv/bin/python scripts/live_shadow_run.py --loop --interval-seconds 60 --send-telegram
```

按 `Ctrl + C` 可以停止。
> ⚠️ 这样跑只在“当前窗口”有效，**一关窗口 / 一断开 SSH 就会停**。要让它 7×24 不掉线，请看 **第 12 章用 systemd 常驻**。

### 8.5 空盘卡片必含字段（看到这些就对了）
**空盘 / dry-run / 被拒绝卡**一定会显示下面这些标记，确认系统处于安全状态：

- `runtime_mode=LIVE_SHADOW`

- `real_capital_changed=false`
- `order_id=--`
- `actual_entry_price=--`

## 第 9 章 启动 Telegram 操作台（接收你的指令）

第 8 章是系统“往外发”信号。本章是让系统能“收”你发的指令（比如 `/status`、`/mode` 等）。

操作台靠 `--poll`（轮询）来不断接收你在 Telegram 里发的命令。**它要一直开着**，你的指令才会被处理。

### 9.1 先在本机测试指令（不连网，验证逻辑）
```bash
set -a; . ./.env; set +a
.venv/bin/python scripts/live_telegram_operator.py --command "/status"
```

### 9.2 启动轮询，开始真正接收 Telegram 指令
```bash
.venv/bin/python scripts/live_telegram_operator.py --poll
```

现在去你的 Telegram，给机器人发 `/help`，它会回给你一份完整指令清单。
按 `Ctrl + C` 停止轮询。（同样建议用 systemd 常驻，见第 12 章。）

### 9.3 常用 Telegram 指令一览

| 指令 | 作用 |
| --- | --- |
| `/help` | 显示所有指令 + 当前模式 |
| `/status` | 当前模式、资金档位、各安全开关、持仓数、净值 |
| `/mode` | 显示当前模式 + 是否允许真实下单 |
| `/positions` | 查看持仓 |
| `/pnl` | 盈亏（毛利、手续费、资金费、净利分开列） |
| `/risk` | 风控档位与亏损状态 |
| `/capital` | 钱包/净值/出入金 |
| `/profile` | 当前资金档位 + 推荐档位 |
| `/pause` | 暂停开新仓（已有仓位不强平） |
| `/resume` | 恢复 |
| `/mode live_limited` | **申请**切到实盘（只返回一个确认码，并不会立刻切、更不会下单） |

> 🔒 安全提醒：只有 `.env` 里 `AMA_TELEGRAM_ALLOWED_CHAT_IDS` 名单内的人发的指令才会被执行，其他人一律拒绝。

## 第 10 章 用 Telegram 指令切换空盘 / 实盘

这正是你要的“**在 Telegram 上一键切换空盘 / 实盘**”。切换采用**双重握手**，防止手滑误切。

### 10.1 从空盘切到实盘（两步握手）

**第 1 步**：在 Telegram 给机器人发：
```
/mode live_limited
```
机器人会回一张卡片，里面有一个**确认码**，长得像 `LIVE-0C58CB6C`，并附上风险摘要（资金档位、最大仓位、最大杠杆等）。此时**模式还没变**。

**第 2 步**：在 **5 分钟内**，把确认码填进去发回：
```
/confirm_live LIVE-0C58CB6C
```

只有当以下条件**全部满足**，实盘（`LIVE_LIMITED`）才会被“武装”：
- 确认码正确且没过期（5 分钟有效）；
- 当前资金档位允许真实下单（即不是 `L0_SHADOW`，要先用 `/profile set L1_10U_PROBE confirm` 切到 10U 档）；
- 紧急刹车没有处于激活状态。

### 10.2 从实盘切回空盘（一步到位）
```
/mode shadow
```
立刻切回最安全的空盘模式，并**解除 LIVE_LIMITED 武装**。**任何时候觉得不对，就发这一句。**

### 10.3 Telegram 切换指令的准确含义（务必读懂）
- `/mode`：只是**查看**当前模式，不改变任何东西。
- `/mode shadow`：切回空盘跑，并解除 LIVE_LIMITED 武装。
- `/mode live_limited`：**只生成一个确认码**，不直接下单、也不立刻切换。
- `/confirm_live CODE`：**只把 LIVE_LIMITED 武装起来**，这不代表会立刻下单。
- **Telegram 永远不能绕过 LiveRiskDecision（风险决策）/ LiveExecutionGateway（执行网关）。**
- **Telegram 永远不能直接裸调币安 API 下单。**
- 真实订单必须走受控的执行路径（详见第 11 章）。

### 10.4 重要事实：切到实盘 ≠ 马上会自动下单
**「武装 `LIVE_LIMITED`」只是打开了第一道门，离真正下单还差好几把钥匙**（见第 11 章）。这是系统刻意设计的“防手滑、防误操作”保险。

## 第 11 章 实盘下单（真金白银，层层确认）

> ⚠️⚠️⚠️ 本章开始会用到真实资金。请务必先用 **10 美元档位** 跑通，确认无误再考虑加大。**亏损风险自负。**

### 关于“全自动实盘”的真相（务必读懂）

本项目出于安全设计，**没有提供“开机即无人值守、自动连环下单”的一键开关**。真实订单只能通过**受控的、需要逐笔确认的下单通道**发出。也就是说：

- 你可以让“空盘信号循环 + Telegram 操作台”长期自动运行（第 8、9、12 章）；
- 但**“按下真实买单”这一步，系统要求人工确认指令 / 确认码**；
- 紧急情况下，平仓也可能需要你**亲自到币安 App 手动操作**。

这是项目作者刻意保留的安全闸门，目的是防止程序在你不知情时把钱亏光。**请按这个事实来规划你的操作，不要指望全程无人值守自动炒币。**

### 11.1 在币安创建 API Key（关键：不要开提现权限）
1. 登录币安网页版 → 头像 → **API 管理** → 创建 API。
2. 权限设置：
   - ✅ 勾选 **读取（Reading）**
   - ✅ 勾选 **允许合约（Enable Futures）**（本系统做 USDT 合约）
   - ❌ **绝对不要勾「允许提现（Enable Withdrawals）」**
3. 如果你的服务器是固定公网 IP，建议在币安里把这把 Key **限制为只允许该 IP 访问**。
4. 把生成的 **API Key** 和 **Secret Key** 保存好（Secret 只显示一次）。

### 11.2 把币安 Key 填进 .env，并打开“私有读”
编辑 `.env`：
```bash
nano .env
```
改成：
```ini
AMA_BINANCE_API_KEY=你的币安APIKey
AMA_BINANCE_API_SECRET=你的币安Secret
AMA_BINANCE_ENABLE_PRIVATE_READ=true
AMA_BINANCE_ENABLE_PRIVATE_TRADE=false   # 这一步先保持 false
```
保存退出，然后重新加载：
```bash
set -a; . ./.env; set +a
```

### 11.3 体检 API 是否连通（只读，不下单）
```bash
.venv/bin/python scripts/live_api_health_check.py --all --json
```
重点看：
- `binance_public_status` 是 `PASS`
- 有私有读 Key 时 `binance_private_read` 是 `PASS`
- `withdraw_permission` 应当是 `False`（**如果是 `True`，回币安把提现权限关掉！**）
- `exchange_live_orders=false`、`ai_trade_authority=false`（正常，安全）

### 11.4 跑“发射前检查”
```bash
.venv/bin/python scripts/live_launch_check.py --binance --telegram --json
```
看 `go_for_live_shadow` 是否为 `true`（空盘就绪）。`go_for_live_limited` 现在大概率是 `false`，因为实盘下单的开关还没全开——这是正常且安全的。

### 11.5 真要实盘下单：开启全部开关
> 只有当你**确实决定**用真钱、并且账户里有 10U 左右时才做这步。

编辑 `.env`：
```bash
nano .env
```
设置：
```ini
AMA_LIVE_CAPITAL_PROFILE_ID=L1_10U_PROBE
AMA_BINANCE_ENABLE_PRIVATE_TRADE=true
AMA_LIVE_EXCHANGE_LIVE_ORDERS=true
AMA_LIVE_TRADE_AUTHORITY=true
AMA_LIVE_AI_TRADE_AUTHORITY=false
AMA_LIVE_EXECUTION_CONFIRM_CODE=自己设一个只有你知道的口令比如 mygo888
```
保存并重新加载：
```bash
set -a; . ./.env; set +a
```

然后在 Telegram 完成第 10 章的握手：
```
/profile set L1_10U_PROBE confirm
/mode live_limited
/confirm_live 返回的LIVE码
```

### 11.6 先空跑一笔（演练，不真下单，并把卡片推到 Telegram）
```bash
.venv/bin/python scripts/live_limited_smoke.py \
  --dry-run \
  --send-telegram \
  --symbol RAVEUSDT \
  --notional 1 \
  --leverage 1 \
  --planned-entry-price 0.5 \
  --planned-stop-price 0.45 \
  --planned-take-profit-price 0.7 \
  --json
```
这会把整条下单链路验证一遍但**不真的下单**。Telegram 应收到一张 `real_order=false` 的模拟订单 / 拦截 / 计划卡，真实成交字段为空白（`order_id=--`、`actual_entry_price=--`、`actual_exit_price=--`、`real_capital_changed=false`）。看输出里 `real_order` 确为 `false`、各项校验是否通过。

### 11.7 真下一笔最小实单（真金白银！）
> 下面这条命令**会真的下一笔约 1 USDT 的单**。确认账户有钱、上面所有开关都开了、确认码对得上，再执行。
> **务必带上止损 / 止盈计划参数，不要在没有止损的情况下下真实单。**

```bash
.venv/bin/python scripts/live_limited_smoke.py \
  --real-order \
  --send-telegram \
  --symbol RAVEUSDT \
  --notional 1 \
  --leverage 1 \
  --planned-entry-price <计划入场价> \
  --planned-stop-price <计划止损价> \
  --planned-take-profit-price <计划止盈价> \
  --i-understand-this-places-real-order \
  --confirm-code <你在11.5设的口令> \
  --json
```

这是真实资金路径。只有**全部条件满足**才会真的成交：`--real-order` + `--i-understand-this-places-real-order` + 确认码正确 + 处于已武装的 `LIVE_LIMITED` + `exchange_live_orders` / `trade_authority` / 私有交易三个开关都为 true + 风险决策 + 执行网关 + 止损/退出计划等全部门控放行。任一不满足，它只会拒绝，**不会乱下单**，并向 Telegram 推一张 `LIVE_EXECUTION_BLOCKED` 拦截卡。

### 11.8 实盘真实成交卡必含字段（成交后核对）
当且仅当**交易所确认成交后**，`LIVE_LIMITED` 的真实成交卡才会出现下面这些字段（之前一律是 `--`）：

- `runtime_mode=LIVE_LIMITED`
- `mode_display=有资金跑`
- `real_order=true`（**仅在交易所确认下单/成交后**）
- `order_id`（交易所订单号）
- `client_order_id`
- `actual_entry_price`（真实成交入场价）
- `actual_exit_price`（真实成交出场价）
- `fee`（手续费）
- `funding`（资金费）
- `gross_pnl`（毛盈亏）
- `net_pnl`（净盈亏 = 毛盈亏 − 手续费 + 资金费）
- `balance_after`（若可得，成交后余额）

下单后请同时在 Telegram 看成交卡（`LIVE_ORDER_SUBMITTED` / `LIVE_ORDER_FILLED` / `LIVE_EXIT_FILLED`，或被挡时的 `LIVE_EXECUTION_BLOCKED`）并回币安 App 核对。

> 💡 **关于 1 USDT 名义额**：1 USDT 可能低于币安某些合约的最小名义额（minNotional），若被交易所或风控拒绝，**只能在当前资金档位（active profile）的上限内调大名义额**，**绝不能绕过风控**。

### 11.9 紧急刹车 / 收尾
- Telegram 发 `/pause` 停止开新仓。
- Telegram 发 `/kill_all` → 再发 `/confirm_kill KILL码` 激活紧急刹车。
- 想彻底回到安全：Telegram 发 `/mode shadow`，并把 `.env` 里 `AMA_LIVE_EXCHANGE_LIVE_ORDERS`、`AMA_LIVE_TRADE_AUTHORITY`、`AMA_BINANCE_ENABLE_PRIVATE_TRADE` 都改回 `false`。
- ⚠️ 紧急刹车只“禁止开新仓”，**已有持仓可能需要你亲自到币安 App 手动平掉**。

## 第 12 章 让程序 7×24 后台常驻（systemd）

前面用 `Ctrl+C` 能停、关窗口就掉的跑法只适合调试。要让“空盘信号推送”和“Telegram 操作台”长期不掉线、崩溃自动重启，用 Linux 自带的 systemd。

> 下面假设：用户是 `root`，项目在 `/root/Altcoin-Momentum-Agent1.0`。
> 如果你的用户/路径不同，把命令里的 `root` 和路径换掉。

### 12.1 准备一个干净的环境变量文件给 systemd 用
systemd 不喜欢带行内注释的 env 文件。我们直接复用 `.env`（它每行就是 `KEY=value`，没问题）。确认权限：
```bash
chmod 600 /root/Altcoin-Momentum-Agent1.0/.env
```

### 12.2 创建“空盘信号推送”服务
```bash
nano /etc/systemd/system/ama-shadow.service
```
粘贴以下内容（原样粘贴即可）：
```ini
[Unit]
Description=AMA-RT LIVE_SHADOW signal pusher (kong pan)
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
WorkingDirectory=/root/Altcoin-Momentum-Agent1.0
EnvironmentFile=/root/Altcoin-Momentum-Agent1.0/.env
ExecStart=/root/Altcoin-Momentum-Agent1.0/.venv/bin/python scripts/live_shadow_run.py --loop --interval-seconds 60 --send-telegram
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```
保存退出（`Ctrl+O` 回车，`Ctrl+X`）。

### 12.3 创建“Telegram 操作台”服务
```bash
nano /etc/systemd/system/ama-operator.service
```
粘贴：
```ini
[Unit]
Description=AMA-RT Telegram operator console (poll)
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
WorkingDirectory=/root/Altcoin-Momentum-Agent1.0
EnvironmentFile=/root/Altcoin-Momentum-Agent1.0/.env
ExecStart=/root/Altcoin-Momentum-Agent1.0/.venv/bin/python scripts/live_telegram_operator.py --poll
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```
保存退出。

### 12.4 启动并设为开机自启
```bash
systemctl daemon-reload
```
```bash
systemctl enable --now ama-shadow.service ama-operator.service
```

### 12.5 查看运行状态和日志
看是否在跑（应显示 `active (running)`）：
```bash
systemctl status ama-shadow.service
systemctl status ama-operator.service
```
实时看日志（按 `Ctrl+C` 退出查看，不影响程序运行）：
```bash
journalctl -u ama-shadow.service -f
```
```bash
journalctl -u ama-operator.service -f
```

### 12.6 停止 / 重启 / 改完配置后生效
```bash
systemctl restart ama-shadow.service     # 重启
systemctl stop ama-shadow.service         # 停止
```
> 📌 你每次改了 `.env` 里的值，要 `systemctl restart` 对应服务才会生效。

## 第 13 章 常见问题排错（看不懂报错先来这）

**Q1：`add-apt-repository ppa:deadsnakes` 报错 / 装不上 Python 3.11？**
改用 pyenv 安装：
```bash
curl -fsSL https://pyenv.run | bash
export PATH="$HOME/.pyenv/bin:$PATH"
eval "$(pyenv init -)"
pyenv install 3.11.9
pyenv local 3.11.9
python -m venv .venv
```
之后照常 `.venv/bin/pip install -r requirements.txt`。

**Q2：运行报错 `SyntaxError` 或 `TypeError: unsupported operand ... | ...`？**
说明你用的 Python 太老（3.9/3.10）。确认虚拟环境是 3.11 建的：
```bash
.venv/bin/python --version   # 必须是 3.11 或更高
```
不对就删掉重建：`rm -rf .venv && python3.11 -m venv .venv && .venv/bin/pip install -r requirements.txt`

**Q3：Telegram 收不到消息 / `--send-test` 显示 `suppressed`？**
逐项检查：① `.env` 里 `AMA_TELEGRAM_OUTBOUND_ENABLED=true`；② Token 填对；③ Chat ID 填对；④ **你本人先给机器人发过一句话**（第 6.3 节）；⑤ 改完 `.env` 记得 `set -a; . ./.env; set +a` 重新加载（或 `systemctl restart`）。

**Q4：Telegram 指令没反应 / 回 `unauthorized`？**
① “操作台”要在跑（`scripts/live_telegram_operator.py --poll` 或 `ama-operator` 服务）；② 你的 Chat ID 必须在 `AMA_TELEGRAM_ALLOWED_CHAT_IDS` 名单里。名单为空 = 拒绝所有人（这是安全设计）。

**Q5：`live_api_health_check` 里 `binance_public_status` 不是 PASS？**
多半是服务器所在地区被币安限制。换香港/新加坡/日本等机房的服务器。

**Q6：`withdraw_permission` 显示 `True`，怎么办？**
你的币安 API Key 开了提现权限。立刻回币安 API 管理里**关闭提现权限**（本系统永远不需要提现）。

**Q7：`/confirm_live` 总是失败？**
① 确认码 5 分钟内有效，过期要重新 `/mode live_limited` 拿新码；② 资金档位要先 `/profile set L1_10U_PROBE confirm`（`L0_SHADOW` 不允许实盘）；③ 紧急刹车若处于激活状态会拒绝，先解除。

**Q8：怎么彻底确认现在没在用真钱？**
Telegram 发 `/status` 看 `runtime_mode`，是 `LIVE_SHADOW` 就绝对安全；或命令行：
```bash
.venv/bin/python scripts/live_telegram_operator.py --status-json | grep -E "runtime_mode|exchange_live_orders|capital_profile"
```

**Q9：运行 `run_paper_cloud` 时报错拒绝启动？**
那是另一套“离线纯模拟验收”流程，它会**拒绝在有币安/TG 真实密钥的环境里启动**。你做信号推送用第 8 章的 `live_shadow_run` 即可，不用管 `run_paper_cloud`。如果想跑它，请到一个**没有设置 `AMA_BINANCE_API_KEY` 等密钥**的干净窗口里运行。

**Q10：怎么判断“当前只是基础状态推送，不是策略候选推送”？**
这是新手最容易误会的点。分三种情况看：

- 如果 Telegram 里只看到这些卡，说明**基础空盘状态推送一切正常**（来源 A）：
  - `LIVE_ACCOUNT_STATUS`
  - `LIVE_POSITIONS`
  - `LIVE_PNL_SUMMARY`
  - `FUNDING_EVENT_SUMMARY`
  - `LIVE_SHADOW_SUMMARY`
- 如果你想**测试模拟开单卡能否推送**（来源 B）：运行 `live_limited_smoke.py --dry-run --send-telegram ...`（见 8.3）。
- 如果你想**确认策略候选链路是否产出候选**（来源 C）：查看日志里是否出现：
  - `SHADOW_ENTRY_PLAN` / `SHADOW_EXIT_PLAN` / `SHADOW_RISK_REJECT`
  - `LIVE_RISK_REJECT` / `LIVE_EXECUTION_BLOCKED`
  - `candidate` / `opportunity_score` / `symbol=XXXUSDT`
  - 看日志命令（systemd 常驻时）：`journalctl -u ama-shadow.service -f`

  没有候选卡 ≠ 出错；很多时候只是当前行情没有满足策略条件的候选。
